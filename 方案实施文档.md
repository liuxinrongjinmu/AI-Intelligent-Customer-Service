## 项目范围界定

本项目为**纯后端 API 服务**，分工如下：

| 范围 | 责任方 | 说明 |
|------|:---:|------|
| 后端 API 设计、开发、测试、部署 | **我方** | 提供标准化 REST API |
| 商家侧知识库管理前端 | 聚宝赞端 | 调用我方管理 API |
| 消费者侧聊天交互前端 | 聚宝赞端 | 调用我方聊天 API（SSE 流式） |
| 内网 Gateway 网关 | 聚宝赞端 | 统一认证与转发 |

`frontend/` 目录下的消费者聊天页面仅供开发阶段本地调试使用，不作为交付物。

### 1.1 技术栈

| 层级 | 技术 | 选型理由 |
|------|------|---------|
| **后端框架** | FastAPI | 原生异步、SSE支持、自动OpenAPI文档、LangChain生态兼容 |
| **Agent编排** | LangGraph | 有状态图编排，条件路由，支持checkpointer持久化 |
| **RAG框架** | LangChain | 完整的文档加载→切片→向量化→检索链路 |
| **LLM** | deepseek-chat | 性价比高，OpenAI兼容API，128K上下文窗口 |
| **Embedding** | BAAI/bge-small-zh-v1.5 | 中文优化，本地运行，~100MB |
| **向量数据库** | ChromaDB 1.5+ | 轻量嵌入式，持久化，支持元数据过滤 |
| **业务数据库** | PostgreSQL 16 | 生产级关系型数据库，支持全文检索、JSONB、并发 |
| **缓存/限流** | Redis 7.4+ | 限流计数、意图缓存、会话缓存 |
| **服务注册发现** | Nacos | 与聚宝赞端统一服务治理，支持服务注册/心跳/发现 |
| **前端** | Jinja2 + 原生JS（SSE） | 复用原型稿风格，服务端渲染 |
| **环境** | Windows + Python 3.11+ | 团队统一 |

```

│ 1. RequestIDMiddleware: 注入 X-Request-ID                    │
│ 2. RateLimitMiddleware: 检查限流（chat: 60次/分）             │
│ 3. 多租户中间件: URL解析 tenant_id，验证 Gateway 认证                    │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─── chat.py: chat_stream() ──────────────────────────────────┐
│ 1. 接收 POST Body: {"message": "查一下我的订单",             │
│                     "thread_id": "xxx", "user_id": "xxx"}    │
│ 2. 构建 graph 输入 state（含 tenant_id, user_id 等）         │
│ 3. 调用 graph.astream_events() → 返回 SSE StreamingResponse │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─── LangGraph Agent 状态流转 ────────────────────────────────┐
│ [classify_intent]                                             │
│   输入: messages + "查一下我的订单" + tenant_id + user_id     │
│   LLM: deepseek-chat, temp=0, json_mode                       │
│   输出: {"intent": "order_query", "sub_type": "order_status", │
```

## 三、核心模块详细设计

### 3.1 AgentState 定义

```python
# backend/agent/state.py
from typing import TypedDict, Annotated, Sequence, NotRequired
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages


class AgentState(TypedDict):
    """
    Agent 图状态定义，在 LangGraph 所有节点间流转
    """
    tenant_id: str                              # 租户ID（每轮从请求注入）
    tenant_name: str                            # 商家名称（用于提示词）
    user_id: str                                # 消费者用户ID
    user_name: str                              # 消费者姓名
    channel: str                                # 来源渠道（web/app/mini_program）
    thread_id: str                              # 会话ID
    messages: Annotated[Sequence[BaseMessage], add_messages]  # 对话历史
    intent: NotRequired[str]                    # 当前轮次识别出的意图大类
    intent_sub_type: NotRequired[str]           # 意图子类
    intent_priority: NotRequired[int]           # 意图优先级
    intent_entities: NotRequired[dict]          # 意图实体抽取结果
    coref_resolved: NotRequired[str]            # 指代消解后的消息
    search_query: NotRequired[str]              # 优化后的检索查询
    suggested_kb_types: NotRequired[list[str]]  # 建议的知识库类型
    retrieved_docs: NotRequired[list]           # 检索召回的知识片段
    final_answer: NotRequired[str]              # 生成的最终回答
    user_profile: NotRequired[dict]             # 用户画像（会员等级/积分/地址）
    recent_orders: NotRequired[list]            # 最近订单列表
    current_order_id: NotRequired[str]          # 当前操作的订单ID
    current_product_id: NotRequired[str]        # 当前查询的商品ID
    ai_failed_count: NotRequired[int]           # AI连续失败计数
    pending_confirmation: NotRequired[dict]     # 待确认操作
```

### 3.2 意图分类体系（INTENT_HIERARCHY）

系统支持 **10 个意图大类 + 42 子类**，按优先级从高到低路由：

| 优先级 | 意图大类 | 子类数量 | 说明 | 工具链 |
|--------|---------|---------|------|--------|
| 1 | `human_service` | 5 | 转人工（用户要求/投诉/情绪/敏感操作/AI失败） | transfer_to_human |
| 3 | `order_query` | 5 | 订单查询（状态/明细/取消/历史/未付款） | query_order |
| 3 | `logistics_query` | 4 | 物流查询（进度/预计送达/改地址/快递公司） | query_order + query_logistics |
| 3 | `product_query` | 5 | 商品咨询（价格/库存/规格/推荐/对比） | query_product |
| 3 | `coupon_query` | 4 | 优惠券咨询（可用券/规则/不可用原因/过期） | query_coupon |
| 3 | `account_query` | 4 | 账户查询（会员等级/积分/地址/安全） | query_user_profile |
| 4 | `knowledge_query` | 7 | 通用知识（售后政策/支付/配送/会员/促销/平台规则） | search_knowledge |
| 5 | `greeting` | 4 | 闲聊问候（打招呼/感谢/告别/无关话题） | 无 |
| 5 | `feedback` | 2 | 反馈确认（肯定/建议） | 无 |
| 5 | `other` | 2 | 兜底（模糊/无法识别） | 无 |

### 3.3 节点函数设计

```python
# backend/agent/nodes.py

def classify_intent_node(state: AgentState) -> dict:
    """
    意图识别节点（含指代消解、实体抽取、知识库类型推荐）
    输入: state.messages + tenant_id + user_id
    处理: 调用 deepseek-chat 做结构化分类
    输出: {"intent": "order_query", "intent_sub_type": "order_status",
           "intent_entities": {...}, "search_query": "...", ...}
    """

def retrieve_knowledge_node(state: AgentState) -> dict:
    """
    知识检索节点（异步，使用 asyncio.to_thread 包装同步检索）
    输入: state（search_query + suggested_kb_types + tenant_id）
    处理: 混合检索（向量 + 关键词 + 租户隔离）
    输出: {"retrieved_docs": [doc1, doc2, ...]}
    """

def generate_answer_node(state: AgentState) -> dict:
    """
    回答生成节点（含 Token 预算管理、流式输出）
    输入: state（messages + retrieved_docs + tenant_name + user_profile）
    处理: 构建系统提示词 → 拼接对话历史 → 流式调用 LLM
    输出: {"final_answer": "...", "messages": [AIMessage(...)]}
    """

def greeting_answer_node(state: AgentState) -> dict:
    """
    问候/反馈/兜底处理节点（统一处理 greeting/feedback/other 三类意图）
    输入: state（当前消息 + intent）
    输出: 友好回复或引导转人工
    """

def order_query_node(state: AgentState) -> dict:
    """
    订单+物流查询节点（logistics_query 也路由到此，统一处理）
    输入: state（intent_entities + user_id + tenant_id）
    处理: 通过 Nacos 发现 tenant-service → 调用聚宝赞订单API
    输出: {"final_answer": "订单信息...", "recent_orders": [...]}
    """

def product_query_node(state: AgentState) -> dict:
    """
    商品查询节点（含知识库降级）
    输入: state（intent_entities + tenant_id）
    处理: 优先调用聚宝赞商品API → 不可用时降级为知识库检索
    输出: {"final_answer": "商品信息..."}
    """

def coupon_query_node(state: AgentState) -> dict:
    """
    优惠券查询节点
    输入: state（user_id + tenant_id）
    处理: 通过 Nacos 发现 coupon-service → 调用聚宝赞优惠券API
    输出: {"final_answer": "优惠券信息..."}
    """

def account_query_node(state: AgentState) -> dict:
    """
    账户查询节点（会员等级/积分/地址）
    输入: state（user_id + tenant_id）
    处理: 通过 Nacos 发现 user-service → 调用聚宝赞用户画像API
    输出: {"final_answer": "账户信息...", "user_profile": {...}}
    """

def complaint_node(state: AgentState) -> dict:
    """
    投诉处理节点
    输入: state（当前消息 + tenant_id）
    处理: 记录投诉信息 + 自动创建转人工工单
    输出: {"final_answer": "投诉已记录..."}
    """

def human_service_node(state: AgentState) -> dict:
    """
    转人工服务节点
    输入: state（intent_sub_type + tenant_id + user_id）
    处理: 创建转人工工单（handoff_service），返回转接话术
    输出: {"final_answer": "正在为您转接..."}
    """
```

### 3.4 LangGraph 图拓扑

```python
# backend/agent/graph.py

def build_graph() -> StateGraph:
    """构建 Agent 图"""
    graph = StateGraph(AgentState)

    # 注册所有节点（10个）
    graph.add_node("classify_intent", classify_intent_node)
    graph.add_node("retrieve_knowledge", retrieve_knowledge_node)
    graph.add_node("generate_answer", generate_answer_node)
    graph.add_node("greeting_answer", greeting_answer_node)
    graph.add_node("order_query_node", order_query_node)
    graph.add_node("complaint_node", complaint_node)
    graph.add_node("human_service_node", human_service_node)
    graph.add_node("product_query_node", product_query_node)
    graph.add_node("coupon_query_node", coupon_query_node)
    graph.add_node("account_query_node", account_query_node)

    graph.add_edge(START, "classify_intent")

    # 条件路由：按优先级分发
    graph.add_conditional_edges("classify_intent", route_by_intent, {
        "retrieve_knowledge": "retrieve_knowledge",
        "order_query_node": "order_query_node",
        "complaint_node": "complaint_node",
        "human_service_node": "human_service_node",
        "greeting_answer": "greeting_answer",
        "product_query_node": "product_query_node",
        "coupon_query_node": "coupon_query_node",
        "account_query_node": "account_query_node",
    })

    graph.add_edge("retrieve_knowledge", "generate_answer")
    # 所有终端节点 → END
    for node in ["generate_answer", "greeting_answer", "order_query_node",
                 "complaint_node", "human_service_node", "product_query_node",
                 "coupon_query_node", "account_query_node"]:
        graph.add_edge(node, END)

    return graph
```

### 3.5 多轮记忆机制

#### 短期记忆：AgentState.messages

每轮对话自动追加到 `messages` 列表（通过 `add_messages` reducer）。生成答案时通过 Token 预算管理器自动裁剪历史。

### 3.6 Token 预算管理

系统在生成回答前自动计算 Token 预算分配：

| 预算项 | 默认值 | 说明 |
|--------|-------|------|
| 总预算 | 8000 tokens | 模型上下文窗口的安全上限 |
| 系统提示词 | 1500 tokens | 固定预算 |
| 对话历史 | 2500 tokens | 动态裁剪，最少保留6轮 |
| 知识上下文 | 2500 tokens | 检索结果自动截断 |
| 响应预留 | 2048 tokens | 确保回答有足够空间 |

当上下文超出预算时，自动裁剪最早的历史消息，保证核心信息不丢失。

#### 服务注册

系统启动时自动注册到 Nacos，服务名 `kefu-service`，并启动心跳循环（每5秒发送心跳）。

```python
# backend/nacos/registry.py
SERVICE_NAME = "kefu-service"

async def register_service() -> bool:
    """注册服务到 Nacos"""
    client.add_naming_instance(
        service_name=SERVICE_NAME,
        ip=ip, port=SERVICE_PORT,
        group_name=NACOS_GROUP,
        healthy=True, enable=True,
        metadata={"version": "1.0.0", "language": "python"},
    )

async def heartbeat_loop():
    """每5秒发送心跳保活"""
    while True:
        client.send_heartbeat(SERVICE_NAME, ip, SERVICE_PORT, ...)
        await asyncio.sleep(5)
```

## 八、关键风险与对策

| 风险 | 可能性 | 影响 | 对策 |
|------|-------|------|------|
| deepseek-chat API 不稳定 | 中 | 对话中断 | 已实现重试机制（2次），超时降级提示 |
| LLM 意图识别错误 | 中 | 路由到错误节点 | 连续2次失败自动转人工 |
| Nacos 服务不可用 | 低 | 业务API调用失败 | 降级为知识库检索 + LLM生成回答 |
| 聚宝赞业务 API 不可用 | 中 | 订单/物流查询失败 | 已实现降级逻辑（知识库兜底 + LLM生成） |
| ChromaDB 多租户隔离漏洞 | 低 | 数据泄露 | 每个查询强制带 `where tenant_id` 过滤 |
| Prompt 注入攻击 | 中 | 系统提示词泄露 | 20种注入模式检测 + 输出过滤 |
| 敏感词绕过 | 低 | 违规内容展示 | 输入+输出双层过滤 |
| LangGraph 版本 API 变化 | 中 | 编译/运行错误 | 锁定 `langgraph>=0.2.1,<1.1` |
| SSE 连接中断 | 中 | 用户看到不完整回复 | 前端断线重连，后端保留对话上下文 |
| 高并发限流 | 中 | 正常用户被拦截 | 分桶限流（chat 60次/分，api 120次/分） |
| SqliteSaver 并发写冲突 | 低 | 对话状态丢失 | WAL模式 + busy_timeout=5000ms |

## 十一、附录：LangGraph 关键技术说明

### 11.1 为什么用 LangGraph 而不是 LangChain 的简单 Chain

| LangChain Chain | LangGraph |
|----------------|-----------|
| 线性 DAG，无分支 | 支持条件分支和循环 |
| 无内置状态管理 | `AgentState` + `AsyncPostgresSaver` |
| 难以实现动态路由 | `conditional_edges` 原生支持 10 种意图路由 |
| 不便于后续扩展 | 插节点不改已有逻辑 |

### 11.2 `astream_events` 事件类型说明

| 事件类型 | 含义 | 使用 |
|---------|------|------|
| `on_chain_start` | 节点开始执行 | 前端显示"正在思考..." |
| `on_chain_end` | 节点执行完毕 | 前端隐藏加载状态 |
| `on_chat_model_stream` | LLM 流式输出 token | **核心**：SSE逐字展现 |
| `on_tool_start` | 工具调用开始 | 业务 API 回调 |
| `on_tool_end` | 工具调用结束 | 业务 API 回调结果 |

### 11.3 后续扩展方向

- 接入更多业务 API（如营销活动、发票管理等）
- 人工坐席工作台（接管/转接/三方会话）
- 多轮对话质量评估
- 知识库自动更新（定时增量同步）
- 性能监控与告警（Prometheus + Grafana）

## 十二、架构优化记录

### 12.1 SQLite → PostgreSQL 全面迁移（2026-06-24）

**迁移目标**：将业务数据库和 LangGraph Checkpoint 数据库从 SQLite 全面迁移至 PostgreSQL，提升并发写入能力和生产环境稳定性。

**修改文件清单**：

| 文件 | 修改内容 |
|------|----------|
| `backend/config.py` | 移除 `SQLITE_PATH`、`CHECKPOINT_PATH`、`ENABLE_FOREIGN_KEYS`；`DATABASE_URL` 改为必填（含默认值） |
| `backend/database.py` | 移除 SQLite 双模式逻辑，PostgreSQL 为唯一引擎；改为延迟初始化避免测试环境导入报错；`SessionLocal`/`engine` 通过 `__getattr__` 向后兼容 |
| `backend/agent/graph.py` | `AsyncSqliteSaver` → `AsyncPostgresSaver`；`aiosqlite` → `langgraph.checkpoint.postgres.aio`；移除 checkpoint SQLite 文件连接逻辑 |
| `backend/utils/backup.py` | 移除 `_backup_sqlite()` 函数和 SQLite 备份分支；移除 checkpoints.db 备份；仅保留 `pg_dump` 备份 |
| `backend/knowledge/sync_log.py` | 移除 `SQLITE_PATH` 导入 |
| `backend/seed.py` | 注释措辞更新（"SQLite" → "数据库"） |
| `deploy.py` | 移除 SQLite 备份命令和清理逻辑 |
| `.env` / `.env.dev` / `.env.prod` | 移除 `SQLITE_PATH`、`CHECKPOINT_PATH`；添加/保留 `DATABASE_URL` |
| `.env.example` | 同上 |
| `docker-compose.yml` | 注释更新（移除 SQLite 回退说明） |
| `requirements.txt` | 移除 `aiosqlite`、`langgraph-checkpoint-sqlite`；添加 `asyncpg`、`langgraph-checkpoint-postgres` |
| `requirements-lock.txt` | 从 ~400 包精简至 ~100 包，移除所有无关依赖 |

**关键设计决策**：
1. `database.py` 采用延迟初始化（`_init_engine()`），避免导入时立即连接数据库，测试环境可安全导入 `Base` 和 `get_db`
2. `SessionLocal` 和 `engine` 通过模块级 `__getattr__` 实现向后兼容延迟属性
3. `AsyncPostgresSaver.from_conn_string()` 使用 psycopg3（非 asyncpg），连接串保持 `postgresql://` 格式
4. 测试 conftest.py 保留内存 SQLite（隔离性好，无需真实 PostgreSQL）

### 12.2 Docker 镜像优化（2026-06-24）

**优化目标**：将 Docker 镜像从 ~9GB 缩减至 2GB 以内。

**优化措施**：

| 优化项 | 具体措施 | 预期效果 |
|--------|----------|----------|
| 依赖精简 | requirements-lock.txt 从 ~400 包精简至 ~100 包 | 减少 ~2GB |
| PyTorch CPU | 通过 `PIP_EXTRA_INDEX_URL` 指定 CPU 版本 | 减少 ~5GB |
| 多阶段构建 | builder 编译 C 扩展 → runtime 仅复制 venv | 减少 ~500MB |
| 缓存清理 | 清理 `__pycache__`、`.pyc`、tests、dist-info | 减少 ~200MB |
| 运行时精简 | 添加 `postgresql-client`（pg_dump 备份用） | 功能完整 |
| .dockerignore | 排除 tests/docs/deploy/.env 等 | 减少构建上下文 |

**Dockerfile 关键变更**：
- Builder 阶段添加 `--no-compile` pip 选项
- Builder 阶段清理 `__pycache__`、`.pyc`、tests 目录、dist-info 元数据
- Runtime 阶段安装 `postgresql-client`（支持 pg_dump 备份）
- COPY 从 `COPY . .` 改为精确复制 `backend/`、`frontend/`、`monitoring/`
