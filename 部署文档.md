## 1. 部署环境准备

### 1.1 服务器配置要求

| 项目 | 最低配置 | 推荐配置 |
|------|---------|---------|
| **CPU** | 2 核 | 4 核 |
| **内存** | 4 GB | 8 GB |
| **磁盘** | 20 GB SSD | 50 GB SSD |
| **操作系统** | Ubuntu 20.04+ / CentOS 7+ | Ubuntu 22.04 LTS |
| **网络** | 可访问聚宝赞内网 VPN | 同左，且可访问 Nacos 和 DeepSeek API |

> **说明**：Embedding 模型（bge-small-zh-v1.5）首次加载时约占用 400MB 内存，运行时约 200MB。ChromaDB 向量库随数据量增长，10 万条向量约占 2GB 磁盘。

### 1.2 依赖软件及版本

| 软件 | 版本要求 | 用途 | 安装方式 |
|------|---------|------|---------|
| **Docker** | ≥ 20.10 | 容器化部署 | 官方仓库安装 |
| **Docker Compose** | ≥ 2.0 | 服务编排 | 随 Docker 安装 |
| **Git** | ≥ 2.30 | 代码拉取 | 系统包管理器 |

> **说明**：本项目采用 Docker 容器化部署，**无需在宿主机安装 Python 或任何 Python 依赖**。所有运行时依赖均在 Docker 镜像内完成安装。

### 1.3 外部服务依赖

| 服务 | 用途 | 需确认事项 |
|------|------|-----------|
| **聚宝赞 Nacos** | 服务注册与发现 | 地址、命名空间 ID、分组名称、认证信息 |
| **聚宝赞 Gateway** | 请求转发 + 认证头注入 | Gateway 注入的验证头名称和值 |
| **聚宝赞 VPN** | 内网安全通道 | VPN 网段 CIDR（用于 IP 白名单） |
| **DeepSeek API** | LLM 大模型调用 | API Key、网络可达性 |
| **聚宝赞 tenant-service** | 业务 API（订单/物流/商品/优惠券/用户） | Nacos 注册的服务名 |

### 1.4 网络端口

| 端口 | 协议 | 用途 | 防火墙策略 |
|------|------|------|-----------|
| **8081** | HTTP | 应用服务端口 | 仅允许 Gateway/VPN 网段访问 |

## 2. 部署步骤

### 2.1 代码拉取

```bash
# 创建项目目录
mkdir -p /opt/kefu-agent && cd /opt/kefu-agent

# 拉取代码（替换为实际仓库地址）
git clone <仓库地址> .

# 切换到目标版本（如有 tag）
git checkout v1.0.0
```

### 2.2 环境变量配置

```bash
# 从模板创建 .env 文件
cp .env.example .env
```

编辑 `.env` 文件，**必须修改**以下配置项：

```bash
# ==================== 必须修改 ====================

# DeepSeek API Key（必填，否则服务无法启动）
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxx

# 管理接口认证密钥（必填，不能使用默认值）
# 生成方式：python -c "import secrets; print(secrets.token_urlsafe(32))"
ADMIN_API_KEY=<生成的高强度密钥>

# Gateway 认证头值（必填，不能使用默认的 "true"）
# 生成方式：python -c "import uuid; print(uuid.uuid4().hex)"
# 需与聚宝赞 Gateway 端约定一致
GATEWAY_VERIFIED_VALUE=<与聚宝赞端约定的令牌>

# Gateway IP 白名单（必填，缩小为 Gateway 实际所在网段）
# 示例：如果 Gateway 在 10.8.0.0/16 网段
GATEWAY_IP_WHITELIST=10.8.0.0/16

# Nacos 服务端地址（必填，指向聚宝赞 Nacos）
NACOS_SERVER_ADDR=<聚宝赞 Nacos 地址>:8848

# Nacos 命名空间 ID（需与聚宝赞端确认）
NACOS_NAMESPACE=<命名空间ID>

# Nacos 分组名称（需与聚宝赞端 Java 端一致）
NACOS_GROUP=DEFAULT_GROUP

# 聚宝赞端在 Nacos 注册的服务名
MERCHANT_SERVICE_NAME=tenant-service

# CORS 允许的来源域名（必填，不能留空）
# 示例：https://chat.jubaozan.com,https://admin.jubaozan.com
ALLOWED_ORIGINS=https://chat.jubaozan.com
```

**可选修改**的配置项：

```bash
# ==================== 可选修改 ====================

# LLM 模型（默认 deepseek-chat）
LLM_MODEL=deepseek-chat

# Embedding 设备（默认 cpu，GPU 服务器可改为 cuda）
EMBEDDING_DEVICE=cpu

# Worker 数量（默认 1，建议设为 CPU 核数的 1-2 倍）
WORKERS=2

# 生产环境关闭 Swagger UI（0=关闭，1=开启）
ENABLE_DOCS=0

# 服务端口（默认 8081）
PORT=8081

# Swagger UI 服务器地址（联调时使用，生产环境关闭 ENABLE_DOCS 即可）
SWAGGER_SERVER_URL=http://<服务器IP>:8081

# Nacos 认证信息（若 Nacos 启用了认证）
NACOS_USERNAME=
NACOS_PASSWORD=

# 我方服务 IP（留空自动获取，Docker 环境通常需要手动指定宿主机 IP）
SERVICE_IP=

# 检索参数（一般无需调整）
RETRIEVAL_TOP_K=5
RETRIEVAL_THRESHOLD=0.2

# Token 预算管理（一般无需调整）
CONTEXT_TOTAL_BUDGET=8000
SYSTEM_PROMPT_BUDGET=1500
HISTORY_MESSAGE_BUDGET=2500
KNOWLEDGE_CONTEXT_BUDGET=2500
RESPONSE_RESERVED_TOKENS=2048
```

### 2.3 构建与启动

```bash
# 构建 Docker 镜像（首次部署或代码更新后执行）
docker compose build

# 启动服务
docker compose up -d

# 查看启动日志，确认无报错
docker compose logs -f kefu-agent
```

**预期日志输出**：

```
INFO:     Started server process [1]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8080 (Press CTRL+C to quit)
INFO:     Nacos 注册成功: kefu-service @ <IP>:8080
```

### 2.4 数据库初始化

```bash
# 在容器内执行种子数据初始化（创建演示租户和示例 FAQ）
docker compose exec kefu-agent python -m backend.seed
```

**预期输出**：

```

创建演示租户...
  tenant_id: demo_001
  API Key:   kefu_xxxxxxxxxxxxxxxx
  (请妥善保存API Key，仅显示一次)
```

> **重要**：API Key 仅在初始化时显示一次，请立即记录保存。后续管理接口调用需要此 Key。

### 2.5 生产租户创建

种子数据创建的是演示租户 `demo_001`。正式使用时需创建生产租户：

## 4. 注意事项

### 4.1 权限设置

```bash
# data 目录需要写入权限（Docker 容器内以 uid=1000 运行）
mkdir -p data
chown -R 1000:1000 data
chmod 750 data

# .env 文件包含敏感信息，限制读取权限
chmod 600 .env
```

### 4.2 防火墙配置

```bash
# 仅允许 VPN/Gateway 网段访问 8081 端口
# 示例（使用 ufw）：
sudo ufw allow from 10.8.0.0/16 to any port 8720 proto tcp
sudo ufw deny 8081/tcp

# 示例（使用 firewalld）：
sudo firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="10.8.0.0/16" port port="8081" protocol="tcp" accept'
sudo firewall-cmd --reload
```

### 4.3 安全策略

| 项目 | 要求 | 说明 |
|------|------|------|
| **ADMIN_API_KEY** | 必须使用高强度随机密钥 | 禁止使用默认值 `change-me-admin-key` |
| **GATEWAY_VERIFIED_VALUE** | 必须使用 UUID 等高强度令牌 | 禁止使用默认值 `true` |
| **GATEWAY_IP_WHITELIST** | 必须缩小为 Gateway 实际网段 | 禁止使用默认全内网段 |
| **ALLOWED_ORIGINS** | 必须配置具体域名 | 禁止留空（留空等价于 `*`） |
| **ENABLE_DOCS** | 生产环境设为 0 | 关闭 Swagger UI，防止接口泄露 |
| **DEEPSEEK_API_KEY** | 仅存储在 .env 文件中 | .env 已被 .gitignore 排除，禁止提交到代码仓库 |
| **.env 文件权限** | 600 | 仅文件所有者可读写 |

### 4.4 数据持久化

Docker Compose 配置了 `./data:/app/data` 卷挂载，以下数据持久化存储在宿主机：

| 路径 | 内容 | 备份建议 |
|------|------|---------|
| `data/chroma_db/` | ChromaDB 向量库（知识库 Embedding） | 知识同步后备份 |
| PostgreSQL 数据 | 业务数据库（租户/会话/反馈/检查点等） | pg_dump 每日备份 |

**备份脚本示例**：

```bash
#!/bin/bash
# 保存为 /opt/kefu-agent/backup.sh，加入 crontab 每日执行
BACKUP_DIR="/opt/kefu-agent/backups/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR"
cp -r data/ "$BACKUP_DIR/"
# 保留最近 7 天备份
find /opt/kefu-agent/backups/ -maxdepth 1 -mtime +7 -exec rm -rf {} +
echo "备份完成: $BACKUP_DIR"
```

```bash
# 加入 crontab（每天凌晨 3 点执行）
crontab -e
0 3 * * * /opt/kefu-agent/backup.sh >> /opt/kefu-agent/backups/backup.log 2>&1
```

### 4.5 资源限制

docker-compose.yml 已配置资源限制：

| 资源 | 上限 | 保留 |
|------|------|------|
| CPU | 2 核 | 0.5 核 |
| 内存 | 2 GB | 512 MB |

如服务器配置较高，可适当调整：

```yaml
deploy:
  resources:
    limits:
      cpus: "4"
      memory: "4G"
    reservations:
      cpus: "1"
      memory: "1G"
```

### 4.6 Worker 数量调整

Worker 数量通过环境变量 `WORKERS` 控制：

| 服务器 CPU 核数 | 建议 Worker 数 | 说明 |
|----------------|---------------|------|
| 2 核 | 1-2 | 每个 Worker 独立进程，注意内存占用 |
| 4 核 | 2-4 | 推荐配置 |
| 8 核+ | 4-6 | 注意 Embedding 模型内存共享 |

> **注意**：多 Worker 模式下，限流计数器各自独立（内存 dict），实际限流阈值为配置值的 N 倍。如需精确限流，需升级为 Redis 方案。

## 5. 常见问题排查

### 5.1 服务启动失败

**症状**：`docker compose up -d` 后容器立即退出

```bash
# 查看退出日志
docker compose logs kefu-agent

# 常见原因及解决
```

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `DEEPSEEK_API_KEY 未配置` | .env 中未设置 API Key | 编辑 .env 填写有效的 DEEPSEEK_API_KEY |
| `Admin API Key 必须配置` | docker-compose 中 ADMIN_API_KEY 未设置 | 在 .env 中设置 ADMIN_API_KEY |
| `Address already in use` | 8081 端口被占用 | 修改 .env 中的 PORT，或停止占用端口的进程 |
| `Permission denied: data/` | data 目录权限不足 | `chown -R 1000:1000 data` |

### 5.2 Nacos 注册失败

**症状**：服务启动正常但 Nacos 控制台看不到服务

```bash
# 检查 Nacos 连通性
docker compose exec kefu-agent python -c "
import httpx
try:
    r = httpx.get('http://<Nacos地址>:8848/nacos/v1/ns/service/list?pageNo=1&pageSize=10')
    print(f'Nacos 状态码: {r.status_code}')
    print(r.text[:200])
except Exception as e:
    print(f'连接失败: {e}')
"
```

| 原因 | 解决方法 |
|------|---------|
| Nacos 地址错误 | 检查 .env 中 NACOS_SERVER_ADDR |
| 网络不通 | 检查 VPN 连接、防火墙规则 |
| 命名空间不存在 | 确认 NACOS_NAMESPACE 与 Nacos 控制台一致 |
| 认证失败 | 填写 NACOS_USERNAME 和 NACOS_PASSWORD |
| SERVICE_IP 不正确 | Docker 环境下需手动指定宿主机 IP |

### 5.3 Gateway 认证失败

**症状**：请求返回 401 Unauthorized

```bash
# 检查请求头
curl -v -H "X-Gateway-Verified: <值>" http://localhost:8720/api/v1/chat/jubaozan_001/stream 2>&1 | grep "< HTTP"
```

| 错误场景 | 原因 | 解决方法 |
|---------|------|---------|
| `Missing X-Gateway-Verified header` | 请求未携带 Gateway 头 | 确认 Gateway 已配置注入验证头 |
| `Invalid gateway verification` | 头值与配置不匹配 | 确认 GATEWAY_VERIFIED_VALUE 与 Gateway 端一致 |
| `IP not in whitelist` | 请求来源 IP 不在白名单 | 检查 GATEWAY_IP_WHITELIST 是否包含 Gateway 网段 |

> **注意**：Docker 容器内看到的客户端 IP 可能是 Docker 网桥 IP（如 172.17.x.x），需确保该 IP 在白名单内，或通过 `X-Real-IP` / `X-Forwarded-For` 头传递真实客户端 IP。

### 5.4 Embedding 模型加载失败

**症状**：首次启动时模型下载超时或失败

```bash
# 查看模型加载日志
docker compose logs kefu-agent | grep -i "embedding\|model\|download"
```

| 原因 | 解决方法 |
|------|---------|
| 网络无法访问 HuggingFace | 配置 HF_ENDPOINT=https://hf-mirror.com（已默认配置） |
| 磁盘空间不足 | 清理磁盘空间，模型约 400MB |
| 内存不足 | 增加服务器内存或 Docker 内存限制 |

> **建议**：首次部署时，Embedding 模型下载可能需要 5-10 分钟（取决于网络速度），期间聊天和知识同步功能不可用，健康检查可能报错。模型下载完成后自动恢复正常。

### 5.5 PostgreSQL 数据库连接失败

**症状**：日志中出现 `DATABASE_URL 未配置` 或连接超时

```bash
# 检查 PostgreSQL 容器状态
docker compose exec postgres pg_isready -U kefu

# 检查连接串配置
docker compose exec kefu-agent env | grep DATABASE_URL
```

| 原因 | 解决方法 |
|------|---------|
| PostgreSQL 容器未启动 | `docker compose up -d postgres` |

### 5.6 ChromaDB 向量库异常

## 6. 回滚方案

### 6.1 版本回滚

```bash
# 1. 停止当前服务
docker compose down

# 2. 切换到上一版本代码
git fetch --tags
git checkout v0.9.0  # 替换为目标版本

# 3. 重新构建并启动
docker compose build
docker compose up -d

# 4. 验证服务正常
curl http://localhost:8720/api/v1/system/health
```

### 6.2 数据回滚

```bash
# 1. 停止服务
docker compose down

# 2. 恢复数据备份（替换为实际备份目录）
cp -r data/ data_failed_$(date +%Y%m%d)/     # 保留当前数据
cp -r /opt/kefu-agent/backups/20260608_030000/data/ data/  # 恢复备份

# 3. 修复文件权限
chown -R 1000:1000 data/

# 4. 启动服务
docker compose up -d
```

### 6.3 紧急降级

如遇严重故障无法快速定位，可执行紧急降级：

```bash
# 1. 停止服务
docker compose down

# 2. 在 Nacos 控制台手动下线服务（防止流量继续路由到本机）

# 3. 通知聚宝赞端切换到降级模式（如备用客服系统）

# 4. 排查问题后重新部署
```

### 6.4 回滚决策标准

| 故障等级 | 表现 | 回滚策略 | 决策时限 |
|---------|------|---------|---------|
| **P0 - 致命** | 服务完全不可用，健康检查失败 | 立即回滚到上一版本 | 5 分钟内 |
| **P1 - 严重** | 核心功能（聊天/同步）不可用 | 尝试快速修复，15 分钟内无果则回滚 | 15 分钟内 |
| **P2 - 一般** | 部分功能异常，不影响核心流程 | 记录问题，下一版本修复 | 不回滚 |
| **P3 - 轻微** | UI 显示异常、日志告警等 | 记录问题，排期修复 | 不回滚 |
